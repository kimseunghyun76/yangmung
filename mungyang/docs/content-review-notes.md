## 검수 범위 및 방법

- 61개 단원(준비 3, 기초 10, 실전 43, 통합 5) 전체를 `lessonForUnit()`이 실제로 렌더링하는 값 기준으로 읽었습니다. 실전 단원은 표현이 난이도 간 누적되는 구조라 고급(advanced) 레벨을 기준으로 검토하고, 입문/중급은 카드 구성만 별도 확인했습니다.
- `src/data/curriculum.ts`, `src/data/lessons.ts`, `src/data/generatedLessons.ts`, `src/data/rewards.ts`를 직접 읽고, 일부는 스크립트로 교차 검증(`requiredIds`와 통합 미션 체크포인트의 `linkedUnitIds` 비교 등)했습니다.
- 검수 기준 7가지: (1) 실제 여행 표현으로서의 실용성 (2) 한국어 해석의 자연스러움 (3) 직원 대사/사용자 대사의 상황 적합성 (4) 단원 간 난이도 곡선 (5) 도감 보상명과 단원 내용의 일치 (6) 중복·유사 단원 여부 (7) 삭제/통합 후보.

## 총평

개별 문장 단위의 일본어·한국어 품질은 전반적으로 양호합니다(`docs/content-rewrite-log.md`에서 이미 한 차례 부자연스러운 표현을 재작성한 흔적이 보이고, 그 결과가 잘 반영되어 있습니다). 다만 **S7~S43(실전 43개 중 37개)이 하나의 자동 생성 템플릿(`generatedLessons.ts`의 `script()` 함수)으로 찍혀 나오면서, 개별 단원을 구별해주는 정보(부제, 학습 팁)가 사실상 소실**되어 있습니다. 이 구조적 문제가 검수 기준 2·4·5에 걸쳐 반복적으로 영향을 주므로 먼저 정리하고, 이후 단원별로 확인된 구체적 문제를 나열합니다.

## A. 구조적 문제 (다수 단원에 공통 적용)

### A1. `subtitle`이 `group`명과 100% 동일 — 실전 43개 단원 전부

S1~S43 전 단원에서 `subtitle` 필드가 `group` 필드와 글자 하나 다르지 않게 동일합니다. 예:

- S1 `title=직원에게 말 걸기 | group=생존 회화와 대화 시작 | subtitle=생존 회화와 대화 시작`
- S24 `title=신칸센 이용 | group=장거리 이동·쇼핑·관광 | subtitle=장거리 이동·쇼핑·관광`
- S43 `title=보안검색과 탑승 | group=출국 | subtitle=출국`

준비/기초(P) 단원과 통합(J) 단원은 `subtitle`에 단원 고유의 설명이 들어가 있는 것과 대조적입니다(예: P1 `subtitle=장음, 촉음, 억양, 빠른 직원 말 듣기`, J1 `subtitle=입국부터 호텔 체크인까지`). 홈/실전 목록 화면에 제목과 부제가 나란히 노출되는 구조상, 사용자에게는 그룹명이 두 번 반복되어 보일 뿐 단원 내용을 미리 짐작할 단서가 없습니다.

- **제안**: 각 단원의 실제 표현 3개를 요약하는 1줄짜리 고유 subtitle을 43개 단원 각각에 부여.

### A2. 학습 팁(`note`/`cue`)이 37개 단원에서 완전히 정형화된 문구로 자동 생성됨

`generatedLessons.ts`의 `script()` 함수가 S7~S43(37개 단원)의 표현 카드를 만들 때, 실제 팁 텍스트를 다음 4개 문구로 기계적으로 채웁니다.

| 카드 | note/cue 문구 (전 단원 동일, {장면}만 치환) |
| --- | --- |
| primary expression | "{장면}에서 먼저 말해야 하는 핵심 표현입니다. 짧게 말하고 직원의 확인 질문을 기다립니다." |
| confirm expression | "상황을 정확히 맞추기 위해 한 번 더 확인하는 표현입니다." |
| support expression | "기본 대화가 끝난 뒤 필요한 정보를 덧붙일 때 씁니다." |
| staffLine | "{장면}에서 직원이 실제로 말할 수 있는 안내입니다. 핵심 명사와 문장 끝을 먼저 잡습니다." |

반면 S1~S6, P1~P13(수작업 단원)은 표현마다 실제로 쓸모 있는 개별 팁을 갖고 있습니다. 예: P5 "右に曲がればいいですか。" 노트 = "안내를 들은 뒤 내가 가야 할 방향을 재확인할 때 씁니다." vs S25 "整理券を取ればいいですか。" 노트 = "상황을 정확히 맞추기 위해 한 번 더 확인하는 표현입니다."(모든 단원 공통 문구) — 정리권을 뽑는 상황 자체에 대한 정보가 전혀 없습니다.

- **영향 단원**: S7, S8, S9, S10, S11, S12, S13, S14, S15, S16, S17, S18, S19, S20, S21, S22, S23, S24, S25, S26, S27, S28, S29, S30, S31, S32, S33, S34, S35, S36, S37, S38, S39, S40, S41, S42, S43 (37개)
- **제안**: 최소한 confirm/support/staffLine의 공통 3문구만이라도 단원별 실제 상황 팁으로 교체. 전수 교체가 부담되면 이용 빈도가 높은 그룹(숙박·식사·문제해결)부터 우선 처리.

### A3. 실전 43개 단원의 선행 조건(`requiredIds`)이 37개 단원에서 완전히 동일

S7부터 S43까지 37개 단원 모두 `requiredIds=P1,P6,P7,P12,P13`로 동일합니다. 즉 이 5개 기초 단원만 마치면 입국심사(S7)든 병원 응급상황(S37)이든 출국 탑승(S43)이든 선행 요건상 차이가 없습니다. 실제 학습 순서는 배열 순서(`nextUnit()`)로만 강제되고, 콘텐츠 의존 관계로는 전혀 게이팅되지 않습니다.

- **제안**: 필수까지는 아니더라도, 그룹 전환 지점(공항→숙박, 숙박→쇼핑, 쇼핑→문제해결)에서 최소 "직전 그룹 절반 이상 완료" 같은 약한 게이팅을 주면 난이도 곡선이 문서(`newbie.md`, `course-restructure.md`)가 원래 의도한 단계적 심화에 더 부합합니다.

### A4. 도감 보상 설명(`description`)이 61개 단원 전부 동일 문구

모든 단원의 보상 설명이 "{단원명} 단원을 완료하면 획득하는 여행 실전 보상입니다."로 고정되어 있습니다. 보상 **이름**(title)은 단원 내용과 대체로 잘 맞습니다만(예: S35 분실물→"분실물 접수표", S43 보안검색과 탑승→"탑승구 안내표"), 설명 문구 자체는 정보값이 없습니다.

- **제안**: 우선순위 낮음. 여유가 되면 보상별로 짧은 플레이버 텍스트 추가.

### A5. P1~P13(13개 단원)이 완전히 동일한 7카드 템플릿을 반복

준비/기초 13개 단원 모두 `preview → expression×3 → staffLine → dialogue → branch → roleplay → quiz → listen → done` 구조를 그대로 반복합니다. 어휘만 바뀔 뿐 카드 흐름·문구 톤이 동일해 실전(S) 단원에 도달하기 전 학습 경험이 다소 단조롭습니다. (버그는 아니며, 검수 기준 4 "난이도 곡선"과 관련된 페이싱 관찰입니다.)

## B. 단원별 구체적 문제

### P1 — 일본어 발음과 듣기 기초

- **문제점**: 이 단원의 목표는 "직원 질문을 듣고 반응하기"(듣기 중심)인데, `expression` 카드 3개(`p1-exp-yoroshii`, `p1-exp-mochikaeri`, `p1-exp-fukuro`)의 실제 문장이 전부 **직원이 손님에게 묻는 존댓말 질문**입니다: "こちらでよろしいですか。"(이쪽으로 괜찮으세요?), "お持ち帰りですか。"(포장이세요?), "袋はご利用ですか。"(봉투 이용하시겠어요?). 그런데 이 카드들이 `staffLine`이 아니라 `expression`(여행자 발화)으로 분류되어, dialogue/roleplay에서 **traveler가 먼저 "이쪽으로 괜찮으세요?"라고 말하는** 것으로 나옵니다. 손님이 직원에게 이 질문을 먼저 던지는 것은 상황상 어색합니다.
- **제안**: 세 문장을 `staffLine`으로 재분류하고, roleplay/dialogue의 traveler 대사는 실제 여행자가 답할 짧은 반응(예: "はい、お願いします。" / "いいえ、大丈夫です。")으로 교체.

### P2 — 여행 가타카나

- **문제점**: roleplay 체크리스트 문구가 `${뜻}라고 먼저 말하기` 형태로 자동 생성되는데, 받침 있는 명사 뒤에도 예외 없이 "라고"가 붙어 "티켓, 승차권라고 먼저 말하기"처럼 조사가 어긋납니다(맞는 표현은 "…승차권이라고"). 또한 `expression` 카드가 완전한 문장이 아니라 단일 명사(チケット/カウンター/レシート)라서, "먼저 말하기 → 확인하기 → 대응하기"라는 역할극 틀 자체가 어색하게 느껴집니다.
- **제안**: 조사 결합 로직에 받침 유무 분기 추가(이/가, 을/를, 라고/이라고 공통 이슈일 가능성 있음, 다른 단원도 전수 확인 권장). 이 단원만이라도 명사 대신 짧은 문장형 예시(예: "チケットをお願いします。")로 교체 검토.

### S9(공항에서 교통편 찾기) / S11(개찰구와 승강장) / S12(환승과 안내방송)

- **문제점**: 세 단원 모두 "역/승강장 위치 확인·환승"이라는 유사한 주제를 각각 3개 문장만으로 다루고 있어, 단원 하나하나의 학습 밀도가 낮고 서로 겹치는 인상을 줍니다. (S9: 전철 타는 곳/리무진버스, S11: 개찰구/플랫폼, S12: 환승/급행 여부 — 모두 "역 안에서 길 찾기"의 하위 항목)
- **제안**: 세 단원을 하나로 합치고 문장 수를 늘리거나, 유지한다면 서로 겹치지 않는 고유 상황(엘리베이터/우선석 문의, 코인로커 등)을 추가해 차별화.

### J1 — 일본 도착일

- **문제점**: `requiredIds=S1,S2,S7,S10,S14`이지만, 실제 체크포인트가 연결하는 단원은 `S7, S8, S9, S10, S14`입니다(`j1-cp2`가 S8, `j1-cp3`가 S9,S10 연결). 즉 **S1·S2는 선행 요건으로 걸려 있지만 미션 안에서 전혀 쓰이지 않고, S8·S9는 미션에서 쓰이지만 선행 요건에는 빠져 있습니다.** 사용자가 S8(수하물과 세관)이나 S9(공항 교통편)를 배우지 않은 상태로도 J1에 진입해 관련 체크포인트를 만날 수 있습니다.
- **제안**: `requiredIds`를 실제 연결 단원 집합인 `S7,S8,S9,S10,S14`로 정정.

### J3 — 지방도시 여행 (cp4 "지역 식당 찾기")

- **문제점**: `j3-cp4` 체크포인트의 목표는 "근처 식당과 이동 가능 여부를 묻는다"인데, 이 체크포인트의 미션 드릴에 표시되는 S39 문장은 "次のバスは何時ですか。"(다음 버스는 몇 시인가요) — `j3-cp2`(지방 교통 환승)와 완전히 동일한 문장입니다. 정작 S39 단원 안에는 체크포인트 목표에 정확히 부합하는 "この辺に食堂はありますか。"(이 근처에 식당이 있나요?)가 이미 존재하는데 쓰이지 않고, 버스 문장이 두 체크포인트에 중복 노출됩니다.
- **제안**: `j3-cp4` 드릴은 S39의 support 표현("この辺に食堂はありますか。")을 사용하도록 연결.

### J5 — 출국일

- **문제점**: `requiredIds=S41,S42,S43`이지만, `j5-cp1` 체크포인트("호텔에서 나가기")는 S17(호텔 서비스와 체크아웃)에 연결되어 있습니다. **S17이 선행 요건에서 빠져 있어**, 체크아웃을 배운 적 없는 사용자도 J5에 진입해 첫 체크포인트를 마주하게 됩니다.
- **제안**: `requiredIds`에 S17 추가 (`S17,S41,S42,S43`).

## C. 참고 사항 (우선순위 낮음, 콘텐츠 품질 자체와는 별개)

- `generatedLessons.ts`의 `DRILL_ONLY_BY_ID`에 `S3` 항목이 정의되어 있으나, 현재 어떤 통합(J) 단원의 체크포인트도 S3를 연결하지 않아 사용되지 않는 데이터로 보입니다. 게다가 이 항목의 문장 구성(出口はどこですか / この道で合っていますか / まっすぐ行って、左ですか / この先を左です)은 실제 `lessons.ts`의 S3 콘텐츠(出口はどこですか / **駅はどちらですか** / この道で合っていますか / この先を左です)와 미묘하게 다릅니다. 나중에 이 항목이 실수로 연결될 경우 S3에서 가르치지 않은 문장이 노출될 수 있습니다.
- `curriculum.ts`의 `LearningUnit.reward`(레거시 문자열 필드)가 61개 단원 전부에 남아 있지만, 실제 화면은 `rewardForUnit()`이 별도 매핑 테이블로 항상 override하므로 이 필드는 화면에 노출되지 않는 죽은 데이터입니다. 콘텐츠 자체의 문제는 아니지만 유지보수 시 혼동 소지가 있습니다.
